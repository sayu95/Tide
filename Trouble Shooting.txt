장고 개발순서
1. Model (데이터 설계)
자바의 Entity: models.py에 테이블 구조를 정의합니다.
python manage.py makemigrations & migrate로 DB에 테이블을 생성합니다.

2. Serializer (DTO 정의)
자바의 Request/Response DTO: serializers.py에 입력 검증 및 출력 형식을 정의합니다.
>> 검증이 필요할때는 serializers
>> 검증이 필요 없을때 dataclass

3. Service (비즈니스 로직)
자바의 Service: service/ 폴더에 실제 로직을 짭니다.
(장고 기본 구조에는 없지만, Slim Serializer 패턴의 핵심!)
User.objects.create_user() 같은 ORM 조작과 예외(raise) 처리를 담당합니다.
방금 UserService 작성을 완료하셨습니다.

4. View (컨트롤러 구현)
자바의 Controller: views.py에서 요청을 받아 서비스를 호출합니다.
시리얼라이저로 데이터를 검증(is_valid)하고 서비스로 넘깁니다.

5. URL (라우팅 설정)
자바의 @RequestMapping: urls.py에 엔드포인트를 등록합니다. (예: path('signup/', views.signup_view))
프로젝트의 urls.py와 앱의 urls.py를 연결합니다.

6. Test (검증)
Postman이나 curl로 API를 쏴서 201 Created나 401 Unauthorized가 잘 나오는지 확인합니다.

장고 쉘(Shell)로 확인 (가장 빠름)
python manage.py shell

영역,데이터 방향,사용하는 객체 (Type),이유
Inbound (입구),Client ➡️ Server,Serializer,"유효성 검사, JSON 파싱, 에러 응답 자동화"
External (수집),외부 API ➡️ Server,dataclass (DTO),고정된 외부 규격을 내부 객체로 가볍게 매핑
Internal (내부),Service ➡️ Model/DB,Model 인스턴스,장고 ORM과의 직관적인 연동
Outbound (출구),Server ➡️ Client,ModelSerializer,DB 객체를 안전하고 규격화된 JSON으로 변환

자바 개발자에게 가장 직관적으로 설명하자면 이렇습니다:
Java의 throw (예외를 발생시킬 때) ➡️ Python의 raise
Java의 throws (메서드 선언부에 붙이는 것) ➡️ Python에는 없음 (생략)

1. throw vs raise (동사: 예외를 던지다)
로직 안에서 "야! 이거 에러야!"라고 선언하는 행위는 완전히 같습니다.
Java: throw new AuthenticationException("Fail");
Python: raise AuthenticationFailed("Fail")
